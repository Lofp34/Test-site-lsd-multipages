# Documentation éditoriale — mode d’emploi

Ce dossier regroupe la documentation de référence pour la production éditoriale du blog Laurent Serre.

Objectif : séparer clairement la **stratégie**, le **fonctionnement du système**, les **garde-fous qualité/diversité**, et le **pilotage vivant des publications**.

## Ordre de lecture recommandé

### 1. `01-editorial-strategy.md`
À lire pour comprendre :
- le positionnement éditorial global,
- les piliers de contenu,
- le rôle du blog dans l’écosystème commercial,
- le ton et la promesse des contenus.

### 2. `02-editorial-operating-system.md`
À lire pour comprendre :
- comment choisir un sujet,
- quelles sources consulter,
- comment scorer et arbitrer,
- comment fonctionne le pipeline éditorial quotidien,
- quelles sorties doivent être produites à chaque publication.

### 3. `03-editorial-guardrails.md`
À lire avant toute nouvelle publication pour éviter :
- les répétitions,
- les angles surreprésentés,
- les redondances avec les derniers articles,
- l’appauvrissement progressif du système éditorial.

### 4. `04-publication-dashboard.md`
Document vivant de pilotage.

Il sert à :
- suivre les publications,
- visualiser les KPI éditoriaux,
- arbitrer les prochaines priorités,
- conserver le backlog et l’historique.

## Logique d’ensemble

La structure repose sur 4 niveaux distincts :

1. **Stratégie** → ce qui est structurel et relativement stable  
2. **Système opératoire** → comment la machine éditoriale fonctionne  
3. **Garde-fous** → ce qui protège la qualité et la diversité  
4. **Pilotage** → ce qui évolue publication après publication

## Principe de gouvernance documentaire

- Si un élément décrit le **positionnement de fond**, il va dans `01-editorial-strategy.md`
- S’il décrit **comment le système choisit, rédige et décline les contenus**, il va dans `02-editorial-operating-system.md`
- S’il décrit une **règle de contrôle, de veto ou de non-répétition**, il va dans `03-editorial-guardrails.md`
- S’il décrit **l’état courant, les publications, les KPI ou le backlog**, il va dans `04-publication-dashboard.md`

## Règle simple

Un document ne doit pas mélanger plusieurs niveaux à la fois.

Autrement dit :
- la vision ne doit pas être noyée dans l’opérationnel,
- l’opérationnel ne doit pas être dispersé dans plusieurs fichiers concurrents,
- les garde-fous doivent être explicites,
- le dashboard doit rester vivant et léger.
